# rc_xiaohou — API 通知系统

一个内部通知投递服务，接收业务系统提交的外部 HTTP 通知请求，并可靠地将其投递到目标地址。

## 问题理解与系统边界

### 这个系统解决什么问题

企业内部多个业务系统在关键事件发生时，需要调用外部供应商的 HTTP API 进行通知（广告回传、CRM 状态更新、库存变更等）。业务系统只关心"通知已提交"，不关心投递结果，也不应当包含重试、超时、供应商格式适配等逻辑。

本服务承担以下职责：

- 统一接收业务系统提交的通知请求（`POST /notifications`）
- 管理供应商 API 配置（URL、Method、认证 Header）
- 在投递到外部 API 时处理失败重试
- 将长期无法投递的通知移入死信队列，防止静默丢失

### 这个系统明确不解决什么

| 问题 | 为什么不在本服务解决 |
|---|---|
| 业务系统的身份认证 | MVP 阶段假设调用方是受信任的内部网络，认证会增加集成成本 |
| 供应商 Body 模板渲染 | 当前版本将 `payload` 原样透传，模板系统可在后续版本按需引入 |
| 投递结果的回调通知 | 业务系统声明不需要关心返回值，回调反而引入耦合 |
| 手动重放死信的 UI | 当前阶段可通过直接查询数据库完成，UI 优先级低于核心可靠性 |
| 分布式部署和水平扩展 | 单进程足以满足 MVP 场景，扩展路径在演进章节说明 |
| 精确一次投递（Exactly-once）| 外部 HTTP API 通常无法提供幂等保证，复杂度不值得 |

## 整体架构

```
业务系统
    │
    │  POST /notifications
    │  { vendor_id, payload }
    ▼
┌─────────────────────────────┐
│         HTTP API Layer      │
│  POST /notifications        │◄──── 立即返回 202 Accepted
│  POST /vendors (管理)        │
│  GET  /notifications/{id}   │
│  GET  /notifications/{id}/attempts │
└──────────────┬──────────────┘
               │ 写入（持久化后才返回 202）
               ▼
┌─────────────────────────────┐
│       SQLite (WAL mode)     │
│  notifications              │
│  delivery_attempts          │
│  vendors                    │
└──────────────┬──────────────┘
               │ 轮询（每 10 秒）
               ▼
┌─────────────────────────────┐
│       Delivery Worker       │
│  - 读取 pending 通知         │
│  - 按供应商配置构建 HTTP 请求  │
│  - 发送到外部 API            │
│  - 记录 delivery_attempts   │
│  - 调度重试 / 移入死信        │
└──────────────┬──────────────┘
               │
               ▼
         外部供应商 API
```

**数据流说明：**

1. 业务系统调用 `POST /notifications`，携带 `vendor_id` 和业务 `payload`
2. API 层验证字段并将通知写入 SQLite（`status=pending`），然后立即返回 `202 Accepted`
3. 后台 Worker 每隔 10 秒轮询 `pending` 状态的通知（批量 claim，标记为 `delivering`）
4. Worker 从 vendors 表读取该供应商的 URL / Method / Headers
5. Worker 发送 HTTP 请求，记录尝试结果
6. 成功 → `status=delivered`；可重试失败 → `status=pending, next_retry_at=T+backoff`；永久失败 → `status=failed`；重试耗尽 → `status=dead`

## 核心设计

### 投递语义：至少一次

本服务选择**至少一次（at-least-once）**投递语义。

**理由：**

- 通知在写入 SQLite 之后才向调用方返回 202，确保已提交的通知不会因进程崩溃而丢失
- 在成功确认前不会删除通知记录，失败后自动重试
- 极端场景（Worker 在收到 2xx 响应后、写 `delivered` 前崩溃）可能导致重复投递
- 接受重复而不接受丢失：业务方的外部 API 收到重复通知，影响是可接受的（幂等或忽略重复）；而丢失通知（用户注册未回传广告、订单未通知库存）则会造成实际业务损失

**与精确一次的对比：**

精确一次需要两阶段提交或外部事务协调器，在 HTTP 交互场景下几乎不可能对外部系统提供真正的 Exactly-once 保证，且实现复杂度与收益不成比例。

### 失败处理策略

#### 重试策略

按以下时间表进行指数退避重试（最多 5 次）：

| 重试次数 | 延迟时间 |
|---------|---------|
| 第 1 次 | 1 分钟  |
| 第 2 次 | 5 分钟  |
| 第 3 次 | 30 分钟 |
| 第 4 次 | 2 小时  |
| 第 5 次 | 8 小时  |

**4xx 不重试**：客户端错误是永久性错误，重试无法解决（错误的 vendor 配置、格式问题），重试只会浪费资源。

**5xx 和网络错误重试**：供应商暂时不可用或网络抖动，期待自愈。

#### 死信队列

重试次数耗尽后，通知状态变为 `dead`（保留在数据库中，不删除）。

**为什么保留而不删除**：静默丢失比数据积累更危险。运维人员可通过查询数据库发现问题并手动处理。

#### 告警（MVP 外，演进方向）

当前版本通过结构化日志（`slog`）记录每次重试和死信事件。生产环境可在此基础上接入告警（见演进章节）。

## 关键工程决策与取舍

### 1. 用 SQLite 代替消息队列

**决策**：用 SQLite 作为持久化队列，而不是 Kafka、RabbitMQ 或 Redis。

**理由**：

- 消息队列解决分布式多消费者场景的问题；单进程内嵌队列不需要外部服务
- SQLite WAL 模式支持并发读，单 Worker 写入性能足够 MVP 场景
- 零运维依赖：无需部署 Broker、维护连接池、处理消息序列化协议
- 迁移路径清晰：`ClaimPendingNotifications` 接口可以在需要时无缝替换为 SQS/Kafka Consumer

**代价**：不适合多实例水平扩展（需要外部协调避免重复消费）。

### 2. 轮询而非事件驱动

**决策**：Worker 每 10 秒轮询数据库，而不是使用 Postgres LISTEN/NOTIFY 或消息队列推送。

**理由**：

- 轮询实现简单，不依赖外部系统的 Push 能力
- 10 秒延迟对通知类场景（非实时）完全可接受
- Postgres LISTEN/NOTIFY 需要换用 Postgres，增加运维成本

**代价**：最坏情况下有 10 秒延迟。如果业务对延迟敏感，可缩短轮询间隔或引入 Push 机制。

### 3. 不做 Body 模板渲染（当前版本）

**决策**：`payload` 原样透传到供应商 API，不做字段映射或模板替换。

**理由**：当前版本专注于可靠投递本身；模板系统是独立的复杂度维度，可按需迭代。

**代价**：业务系统需要自行按供应商格式构造 payload，存在一定耦合。

### 4. 不做调用方认证

**决策**：MVP 阶段不验证业务系统的身份。

**理由**：假设服务部署在内部网络，调用方均为受信任的内部服务。添加认证会增加集成成本，且在初期意义不大。

**代价**：内网任意服务均可调用，需要在网络层做访问控制。

## 未来演进方向

当流量或复杂度显著增长时，推荐以下演进路径：

### 短期（流量 10x）

- **告警集成**：监控 `status=dead` 记录数量，超阈值触发 PagerDuty 或飞书报警
- **死信重放 API**：提供 `POST /notifications/{id}/retry` 端点，运维可手动触发重放
- **可配置重试策略**：支持按 vendor 配置最大重试次数和退避系数

### 中期（多实例部署需求）

- **替换队列后端**：将 `ClaimPendingNotifications` 换为 SQS、RocketMQ 等支持分布式消费的消息队列，Worker 可水平扩展
- **调用方认证**：引入 API Key 或 mTLS，限制调用方

### 长期（高可靠性生产）

- **Body 模板引擎**：支持 Go template 或 JSONata 进行字段映射，解耦业务系统与供应商格式
- **幂等键**：支持调用方传入 `idempotency_key`，服务端去重，将至少一次语义提升为对调用方的幂等保证
- **供应商 SLA 监控**：统计每个供应商的成功率和 P99 延迟，自动熔断表现差的供应商

## 快速启动

```bash
# 运行服务（SQLite 文件默认为 notifications.db）
go run ./cmd/server

# 注册一个供应商
curl -X POST http://localhost:8080/vendors \
  -H "Content-Type: application/json" \
  -d '{"name":"crm","url":"https://crm.example.com/webhook","method":"POST","headers":{"Authorization":"Bearer token"}}'

# 提交一条通知
curl -X POST http://localhost:8080/notifications \
  -H "Content-Type: application/json" \
  -d '{"vendor_id":"<vendor-id>","payload":{"event":"user.subscribed","user_id":"u123"}}'

# 查询投递状态
curl http://localhost:8080/notifications/<notification-id>
curl http://localhost:8080/notifications/<notification-id>/attempts
```

**环境变量：**

| 变量 | 默认值 | 说明 |
|------|-------|------|
| `ADDR` | `:8080` | HTTP 监听地址 |
| `DB_PATH` | `notifications.db` | SQLite 文件路径 |
| `MAX_RETRIES` | `5` | 最大重试次数 |

## AI 使用说明

本节说明 AI 在设计与实现过程中发挥的作用、被采纳与未采纳的建议，以及核心决策的归属。

### AI 在哪些地方提供了帮助

- **项目结构**：AI 提供了 `cmd/server / internal/api / internal/store / internal/delivery` 的分层结构，符合 Go 社区惯例，直接采用
- **SQLite Schema 设计**：AI 给出了 `notifications` 和 `delivery_attempts` 的初始表结构，包括 WAL 模式和 `foreign_keys=ON` 的建议，经审查后保留
- **指数退避代码**：`BackoffSchedule []time.Duration` 的实现由 AI 完成，逻辑清晰，无需修改
- **`ShouldRetry` 逻辑**：AI 提供了按 HTTP 状态码区分重试/不重试的骨架，我在此基础上补充了对网络错误类型的判断

### AI 给出过但没有采纳的建议

| AI 建议 | 未采纳原因 |
|--------|----------|
| 使用 Kafka 实现至少一次投递 | Kafka 解决分布式多消费者问题；单进程场景 SQLite 已经足够，引入 Kafka 会增加大量运维成本 |
| 引入 Redis 作为任务队列 | 同上，增加外部依赖，Redis 故障会导致服务不可用，而 SQLite 内嵌无此风险 |
| 每个供应商独立配置重试策略 | YAGNI——在还不知道各供应商 SLA 差异有多大之前，统一策略更易维护 |
| 引入 OpenTelemetry 做分布式追踪 | MVP 阶段结构化日志（`slog`）已足够定位问题，追踪系统的接入成本在此阶段不值得 |
| 用 Circuit Breaker 防止对不可用供应商持续发送 | 正确的长期方向，但当前退避策略已缓解了这个问题；Circuit Breaker 引入了额外状态，放在下一版本 |
| 使用 Postgres LISTEN/NOTIFY 取代轮询 | 需要切换到 Postgres，增加运维成本；10 秒轮询对当前场景已足够 |

### 哪些关键决策是我自己做出的

**1. SQLite 作为持久化队列的选型**

AI 倾向于推荐"生产级"方案（Kafka、Redis）。我明确选择 SQLite：这不是妥协，而是在系统边界内主动管控复杂度的判断——单进程、单文件，零运维依赖，足以满足 MVP 场景，且迁移路径清晰。

**2. 4xx 错误不消耗重试预算、直接永久失败**

这个行为在 AI 给的初版代码里不存在，我手动加入：4xx（如 404、401）是客户端配置错误，重试不会改变结果，只会浪费时间窗口。正确做法是立即标记 `failed` 并记录，让运维尽快发现配置问题。

**3. 死信保留而不删除**

AI 没有显式讨论死信的处置策略。我的判断是：静默丢失比数据积累更危险。删除死信会让问题不可见；保留让运维有机会排查和手动重放。

**4. 不做调用方认证的边界判断**

这是一个明确的"不做"决策，不是遗漏。我在系统边界中记录了假设（内部受信网络），并说明了代价（需要网络层隔离）。这种取舍说明是有意为之的设计，而不是未完成的功能。
